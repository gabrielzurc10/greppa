# Greppa

Upload a file or folder, scan it for vulnerabilities with [Semgrep](https://semgrep.dev) (via its MCP server), and get an AI-enriched report: a severity chart plus, for every finding, the affected file/line, an explanation of why it matters, and a suggested fix generated by GPT-5-mini.

## Architecture

```
Angular 21 SPA ──────► Azure Static Web Apps (Free tier)
      │ polls
      ▼
ASP.NET (.NET 10) API ─► Azure Container Apps (scale-to-zero, max 1 replica)
      │                        ▲ image from Azure Container Registry (Basic)
      ├─► semgrep MCP server (stdio subprocess, `semgrep mcp`)
      └─► OpenAI API (gpt-5-mini, structured outputs)
```

- **Scan flow**: `POST /api/scans` (multipart upload) returns a job id immediately; the frontend polls `GET /api/scans/{id}` every 2.5 s. A background worker runs semgrep via the **Model Context Protocol** (official `ModelContextProtocol` C# SDK talking to the `semgrep mcp` stdio server), then enriches each finding with **gpt-5-mini** (strict JSON schema output, 4 concurrent calls, graceful fallback to semgrep's message).
- **SOLID backend layering**: `Greppa.Domain` (entities) ← `Greppa.Application` (interfaces + `ScanOrchestrator`) ← `Greppa.Infrastructure` (Semgrep MCP, OpenAI, in-memory job store, channel queue — all behind interfaces) ← `Greppa.Api` (composition root). Every dependency is constructor-injected; `Scanner:UseFake` / `Enricher:UseFake` swap in offline fakes with one registration.
- **Async by necessity**: Container Apps ingress has a hard 240 s timeout; scans + LLM calls can exceed it. `max_replicas = 1` is required because the job store is in-memory.
- **Cold start**: with scale-to-zero, the first request after idle takes ~15–30 s (the image bundles Python + semgrep). The UI shows a "starting the scanner" state.

## Run locally

Prereqs: Docker, Node 24 + Angular CLI, and a `.env` at the repo root:

```
OPENAI_API_KEY=sk-...
SEMGREP_APP_TOKEN=...
```

```bash
# real pipeline: semgrep + gpt-5-mini inside the container
docker compose up --build backend

# frontend with /api proxied to localhost:8080
cd frontend && npm install && npm start   # http://localhost:4200
```

Fast no-Docker loop (fake scanner + fake enricher, no keys needed):

```bash
dotnet run --project backend/Greppa.Api   # Development config uses the fakes
```

Smoke-test the API directly:

```bash
curl -F "files=@some/vulnerable.py" http://localhost:8080/api/scans   # -> { "jobId": "..." }
curl http://localhost:8080/api/scans/<jobId>                          # poll until completed
```

## Deploy to Azure

Everything runs in your existing **`greppa-rg`** resource group (East US; the Static Web App lives in East US 2 because the Free tier isn't offered in East US). Terraform never creates or deletes the resource group itself.

### One-time setup

1. **Create the GitHub repo** and add it as `origin` (don't push yet).
2. **Terraform state storage** (creates `greppa-tfstate-rg` + a private storage account, and rewrites `terraform/backend.tf` with the generated name — commit that change):
   ```bash
   az login
   ./scripts/bootstrap-state.sh
   ```
3. **GitHub → Azure OIDC** (app registration + federated credential + Contributor role; prints the values for the secrets below):
   ```bash
   ./scripts/bootstrap-oidc.sh <github-owner>/<repo>
   ```
4. **GitHub Actions secrets** (Settings → Secrets and variables → Actions):

   | Secret | Value |
   |---|---|
   | `AZURE_CLIENT_ID` | printed by `bootstrap-oidc.sh` |
   | `AZURE_TENANT_ID` | printed by `bootstrap-oidc.sh` |
   | `AZURE_SUBSCRIPTION_ID` | printed by `bootstrap-oidc.sh` |
   | `OPENAI_API_KEY` | from your `.env` |
   | `SEMGREP_APP_TOKEN` | from your `.env` |

   No other credentials are needed: GitHub→Azure auth is OIDC (no client secret), ACR credentials flow through Terraform, and the SWA deployment token is read from Terraform output inside the workflow.

### Deploy

```bash
git push origin main
```

`deploy.yml` then: applies Terraform for the registry → builds & pushes the backend image (tagged with the git SHA) → applies the rest (Container App, Static Web App, Log Analytics) → builds the Angular app, stamps the backend URL into `config.json` → uploads to Static Web Apps. The run summary shows the frontend URL. Re-deploys are just more pushes (or run the workflow manually).

### Destroy

Actions → **Destroy** → Run workflow → type `destroy` in the confirm box. Removes the Container App, ACR, SWA, and Log Analytics but keeps `greppa-rg` and the tfstate storage account, so the next deploy recreates everything cleanly.

## Cost

Roughly **$5–7/month while deployed** — ACR Basic (~$5) is the only fixed cost; Container Apps scale-to-zero stays inside the free consumption grant at portfolio traffic, SWA Free and Log Analytics (5 GB/mo free) are $0, and gpt-5-mini costs ~$0.01–0.05 per scan. Destroyed between demos, idle cost is effectively $0.

## Notable implementation details

- The deprecated `semgrep-mcp` PyPI package is **not** used — the MCP server now ships inside the semgrep binary (`semgrep mcp`, semgrep ≥ 1.16x). Its `semgrep_scan` tool takes absolute file paths; the daemon scans renamed copies, so results are mapped back by basename (batches are built so basenames stay unique per call).
- gpt-5-mini is a reasoning model: no `temperature`; responses are constrained with a strict JSON schema (`reason`, `suggestedFix`).
- Upload limits: 50 MB total, 1 MB/file, 200 files; binaries, `node_modules/`, `.git/`, and lockfiles are skipped server-side.
- Severity mapping: semgrep ERROR→High, WARNING→Medium, INFO→Low, promoted to Critical when rule metadata marks impact **and** likelihood HIGH.
